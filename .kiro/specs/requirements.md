# Requirements Document

## Introduction

The AI Study Companion is a standalone Next.js web application that uses Hindsight persistent
memory to help students overcome the core problem of forgetting what they struggle with. The
companion remembers every quiz attempt, mistake pattern, and study session across time, enabling
it to generate personalized revision questions, adaptive study plans, and smart exam reminders —
all grounded in the student's actual history rather than generic heuristics.

The primary interface is a web application built with Next.js 14 (App Router), TypeScript, and
Tailwind CSS, featuring a chat UI and a dashboard that surfaces weak areas, study history, and
upcoming exams. Hindsight (vectorize-io/hindsight) serves as the persistent memory backend.
The LLM layer is powered by Groq or OpenAI. Optionally, the application can register itself as
an OpenClaw skill so students can also interact via messaging apps (Telegram, WhatsApp, Slack,
Discord, etc.) — but this is a bonus capability, not the core delivery mechanism.

The project lives in its own repository folder (`study-companion`) and is fully self-contained.

## Glossary

- **Study_Companion**: The standalone Next.js web application that acts as the student's AI
  study assistant.
- **Web_UI**: The Next.js front-end providing the chat interface and dashboard.
- **Chat_Interface**: The conversational UI component within Web_UI where the Student sends
  messages and receives responses.
- **Dashboard**: The Web_UI view that displays Weak_Areas, study history, and upcoming exams at
  a glance.
- **Hindsight**: The persistent vector memory system (vectorize-io/hindsight) used to store and
  retrieve all student history.
- **Memory_Store**: The Hindsight instance that persists quiz results, mistake patterns, and
  study sessions for a given student.
- **Quiz_Engine**: The sub-component of Study_Companion responsible for generating and evaluating
  quiz questions.
- **Mistake_Tracker**: The sub-component that records, aggregates, and surfaces patterns of
  errors per topic and subject.
- **Schedule_Manager**: The sub-component that records study sessions and manages exam reminders.
- **Plan_Generator**: The sub-component that synthesizes memory data into a personalized study
  plan.
- **Student**: The human user interacting with Study_Companion through the Web_UI or, optionally,
  an OpenClaw channel.
- **Session**: A single continuous interaction between a Student and Study_Companion.
- **Quiz_Record**: A persisted Hindsight memory entry containing question, student answer,
  correct answer, subject, topic, timestamp, and score.
- **Mistake_Pattern**: A persisted Hindsight memory entry summarizing recurring errors for a
  specific subject/topic combination.
- **Study_Session_Record**: A persisted Hindsight memory entry capturing date, duration, subjects
  covered, and self-reported confidence.
- **Exam_Event**: A persisted Hindsight memory entry for an upcoming exam with subject, date, and
  reminder preferences.
- **Weak_Area**: A subject/topic combination where the Student's historical accuracy falls below
  the configured threshold.
- **Revision_Question**: An AI-generated question specifically targeting a Weak_Area.
- **Study_Plan**: A structured, time-boxed schedule of topics and activities generated by
  Plan_Generator based on memory.
- **Confidence_Score**: A 1–5 self-reported rating the Student provides after a study session.
- **Accuracy_Rate**: The percentage of correct answers for a given subject/topic, computed from
  Quiz_Records in Memory_Store.
- **Weak_Area_Threshold**: The configurable Accuracy_Rate below which a topic is classified as a
  Weak_Area (default: 60%).
- **OpenClaw**: Open-source personal AI assistant infrastructure (github.com/openclaw) that
  routes messages from channels like WhatsApp, Telegram, Slack, and Discord to registered skills.
- **OpenClaw_Skill**: An optional adapter that registers Study_Companion with an OpenClaw Gateway
  so Students can interact via messaging apps.

## Requirements

### Requirement 1: Hindsight Memory Integration

**User Story:** As a developer, I want Study_Companion to use Hindsight as its sole persistent
memory backend, so that all student history survives across sessions and the Hindsight usage
criterion is fully satisfied.

#### Acceptance Criteria

1. THE Study_Companion SHALL connect to a Hindsight instance using credentials supplied via
   environment variables at startup.
2. WHEN Study_Companion writes any Quiz_Record, Mistake_Pattern, Study_Session_Record, or
   Exam_Event, THE Study_Companion SHALL persist it to Memory_Store within 2 seconds.
3. WHEN Study_Companion reads student history, THE Study_Companion SHALL retrieve relevant
   memories from Memory_Store using semantic search with the student's query as the search key.
4. IF the Hindsight connection fails at startup, THEN THE Study_Companion SHALL surface a
   descriptive error message in the Web_UI and halt further operations until the connection is
   restored.
5. IF a memory write to Hindsight fails, THEN THE Study_Companion SHALL retry the write up to 3
   times with exponential backoff before surfacing an error to the Student.
6. THE Memory_Store SHALL store each memory entry with a structured metadata envelope containing:
   entry_type, student_id, subject, topic, timestamp, and session_id.
7. FOR ALL memory entries written and then retrieved, THE Study_Companion SHALL produce an entry
   whose subject, topic, and entry_type fields are identical to those written (round-trip
   property).

### Requirement 2: Web UI — Chat Interface and Dashboard

**User Story:** As a student, I want a web-based interface with a chat window and a dashboard, so
that I can interact with the Study_Companion and see my progress at a glance without installing
any app.

#### Acceptance Criteria

1. THE Web_UI SHALL provide a Chat_Interface where the Student can send natural-language messages
   and receive responses from Study_Companion.
2. THE Web_UI SHALL provide a Dashboard that displays: the Student's top Weak_Areas, a summary
   of recent study sessions, and a list of upcoming Exam_Events sorted by date ascending.
3. WHEN a Student sends a message in the Chat_Interface, THE Web_UI SHALL display the
   Study_Companion's response within the same chat thread within 5 seconds of the message being
   sent.
4. THE Web_UI SHALL update the Dashboard automatically after each completed quiz session or
   logged study session without requiring a full page reload.
5. WHEN the Student navigates to the Dashboard, THE Web_UI SHALL retrieve and render the latest
   Weak_Areas, Study_Session_Records, and Exam_Events from Memory_Store.
6. THE Web_UI SHALL be responsive and render correctly on screen widths from 375px (mobile) to
   1440px (desktop).
7. IF the Study_Companion backend returns an error, THEN THE Web_UI SHALL display a descriptive
   inline error message in the Chat_Interface without crashing the page.

### Requirement 3: Quiz Recording and Retrieval

**User Story:** As a student, I want the Study_Companion to remember every quiz I take, so that
it can track my progress and identify where I need more practice.

#### Acceptance Criteria

1. WHEN a Student completes a quiz question, THE Quiz_Engine SHALL persist a Quiz_Record to
   Memory_Store containing: question text, student answer, correct answer, subject, topic,
   timestamp, and a binary correct/incorrect flag.
2. WHEN a Student requests their quiz history for a subject, THE Study_Companion SHALL retrieve
   all Quiz_Records for that subject from Memory_Store and present them in reverse chronological
   order.
3. THE Quiz_Engine SHALL compute the Accuracy_Rate for each subject/topic pair by aggregating all
   Quiz_Records for that pair from Memory_Store.
4. WHEN a Student requests their overall performance summary, THE Study_Companion SHALL present
   Accuracy_Rate values for every subject/topic pair that has at least one Quiz_Record in
   Memory_Store.
5. IF a Student provides an answer to a quiz question, THEN THE Quiz_Engine SHALL evaluate the
   answer and return a correct/incorrect verdict with an explanation within 5 seconds.
6. THE Quiz_Engine SHALL support quiz questions in the following formats: multiple-choice,
   true/false, and short-answer.
7. WHEN a Student requests a quiz on a specific subject, THE Quiz_Engine SHALL generate questions
   that have not appeared in the Student's last 10 Quiz_Records for that subject, to avoid
   repetition.

### Requirement 4: Mistake Pattern Tracking

**User Story:** As a student, I want the Study_Companion to recognize patterns in my mistakes, so
that I can focus my revision on the areas where I consistently go wrong.

#### Acceptance Criteria

1. WHEN a Student answers a quiz question incorrectly, THE Mistake_Tracker SHALL update the
   Mistake_Pattern entry for the corresponding subject/topic in Memory_Store, incrementing the
   error count and appending the incorrect answer.
2. THE Mistake_Tracker SHALL classify a subject/topic as a Weak_Area WHEN the Accuracy_Rate for
   that pair falls below the Weak_Area_Threshold.
3. WHEN a Student requests their Weak_Areas, THE Mistake_Tracker SHALL retrieve all
   Mistake_Pattern entries from Memory_Store and return the subject/topic pairs ranked by error
   count in descending order.
4. WHEN a Student's Accuracy_Rate for a previously classified Weak_Area rises above the
   Weak_Area_Threshold across at least 5 consecutive Quiz_Records, THE Mistake_Tracker SHALL
   update the Mistake_Pattern entry to remove the Weak_Area classification.
5. THE Mistake_Tracker SHALL persist a Mistake_Pattern summary to Memory_Store after every 5
   incorrect answers for a given subject/topic, capturing: total errors, most common wrong
   answers, and the timestamp of the last error.
6. IF a Student has no Quiz_Records in Memory_Store, THEN THE Mistake_Tracker SHALL return an
   empty Weak_Areas list with a message indicating no history is available yet.

### Requirement 5: Study Schedule Memory

**User Story:** As a student, I want the Study_Companion to remember when and what I study, so
that it can build a realistic plan around my actual habits.

#### Acceptance Criteria

1. WHEN a Student logs a study session, THE Schedule_Manager SHALL persist a Study_Session_Record
   to Memory_Store containing: date, start time, duration in minutes, subjects covered, topics
   covered, and Confidence_Score.
2. WHEN a Student requests their study history, THE Schedule_Manager SHALL retrieve all
   Study_Session_Records from Memory_Store and present them grouped by subject in reverse
   chronological order.
3. THE Schedule_Manager SHALL compute the average daily study duration per subject by aggregating
   Study_Session_Records from Memory_Store over the past 30 days.
4. WHEN a Student adds an Exam_Event, THE Schedule_Manager SHALL persist it to Memory_Store with:
   subject, exam date, and reminder lead times (default: 7 days, 3 days, 1 day before).
5. WHEN the current date matches a reminder lead time for an Exam_Event, THE Schedule_Manager
   SHALL display a proactive reminder notification in the Web_UI Dashboard.
6. IF a Student logs a study session with a duration of 0 minutes, THEN THE Schedule_Manager
   SHALL reject the entry and return a validation error message.
7. WHILE an Exam_Event exists in Memory_Store with an exam date in the future, THE
   Schedule_Manager SHALL include the subject in all Study_Plan generation requests.

### Requirement 6: Personalized Study Plan Generation

**User Story:** As a student, I want the Study_Companion to generate a study plan based on my
actual history, so that I spend time on what matters most rather than following a generic
schedule.

#### Acceptance Criteria

1. WHEN a Student requests a study plan, THE Plan_Generator SHALL retrieve Weak_Areas,
   Study_Session_Records, and Exam_Events from Memory_Store before generating the plan.
2. THE Plan_Generator SHALL allocate more study time to subjects with lower Accuracy_Rate values,
   proportional to the inverse of the Accuracy_Rate for each subject.
3. WHEN generating a study plan, THE Plan_Generator SHALL respect the Student's historical
   average daily study duration per subject as retrieved from Memory_Store, and SHALL NOT
   schedule more than 150% of that average per day for any subject.
4. THE Plan_Generator SHALL produce a study plan covering a time horizon specified by the Student
   (minimum 1 day, maximum 90 days).
5. WHEN an Exam_Event is present in Memory_Store, THE Plan_Generator SHALL schedule a dedicated
   revision session for that subject within the 3 days preceding the exam date.
6. THE Plan_Generator SHALL include at least one Revision_Question session per Weak_Area per week
   in the generated study plan.
7. IF a Student has fewer than 3 Quiz_Records in Memory_Store, THEN THE Plan_Generator SHALL
   generate a generic onboarding plan and inform the Student that the plan will personalize as
   more history is recorded.

---

### Requirement 7: AI-Generated Revision Questions Targeting Weak Areas

**User Story:** As a student, I want the Study_Companion to generate revision questions aimed at
my weak areas, so that my practice is targeted and efficient.

#### Acceptance Criteria

1. WHEN a Student requests revision questions, THE Quiz_Engine SHALL retrieve the Student's
   Weak_Areas from Memory_Store and generate Revision_Questions exclusively for those topics.
2. THE Quiz_Engine SHALL generate Revision_Questions that vary in format (multiple-choice,
   true/false, short-answer) across a single revision session.
3. WHEN generating a Revision_Question for a Weak_Area, THE Quiz_Engine SHALL incorporate the
   most common wrong answers from the corresponding Mistake_Pattern in Memory_Store to construct
   plausible distractors.
4. THE Quiz_Engine SHALL generate a minimum of 3 and a maximum of 20 Revision_Questions per
   revision session, with the count configurable by the Student.
5. WHEN a Student completes a revision session, THE Quiz_Engine SHALL persist all answers as
   Quiz_Records to Memory_Store and update the corresponding Mistake_Pattern entries.
6. IF a Student has no Weak_Areas in Memory_Store, THEN THE Quiz_Engine SHALL generate general
   questions across all subjects the Student has previously studied and inform the Student that
   no weak areas have been identified yet.

### Requirement 8: Smart Exam Reminders

**User Story:** As a student, I want the Study_Companion to remind me about upcoming exams at the
right time, so that I never walk into an exam underprepared.

#### Acceptance Criteria

1. WHEN a Student adds an Exam_Event, THE Schedule_Manager SHALL confirm the event is persisted
   to Memory_Store and echo back the exam subject, date, and scheduled reminder times.
2. THE Schedule_Manager SHALL check Memory_Store for due Exam_Event reminders on each page load
   of the Web_UI and SHALL display a reminder notification if the current date matches a
   configured lead time.
3. WHEN displaying an exam reminder, THE Schedule_Manager SHALL include: exam subject, days
   remaining, the Student's current Accuracy_Rate for that subject, and the top 3 Weak_Areas for
   that subject from Memory_Store.
4. WHEN a Student requests a list of upcoming exams, THE Schedule_Manager SHALL retrieve all
   future Exam_Events from Memory_Store and present them sorted by exam date ascending.
5. IF an exam date has passed, THEN THE Schedule_Manager SHALL mark the Exam_Event as completed
   in Memory_Store and SHALL NOT send further reminders for it.
6. WHERE a Student has configured custom reminder lead times for an Exam_Event, THE
   Schedule_Manager SHALL use those lead times instead of the defaults.

### Requirement 9: Multi-Subject and Multi-Topic Support

**User Story:** As a student studying multiple subjects, I want the Study_Companion to track each
subject and topic independently, so that my history and plans are accurate across my full
curriculum.

#### Acceptance Criteria

1. THE Study_Companion SHALL support an unbounded number of subjects and topics per Student,
   limited only by Memory_Store capacity.
2. WHEN a Student queries history, plans, or weak areas, THE Study_Companion SHALL scope all
   Memory_Store retrievals to the requesting Student's student_id to prevent cross-student data
   leakage.
3. THE Study_Companion SHALL allow a Student to list all subjects and topics for which
   Quiz_Records exist in Memory_Store.
4. WHEN a Student renames a topic, THE Study_Companion SHALL update all existing Memory_Store
   entries for that topic to use the new name within a single operation.
5. IF a Student requests data for a subject that has no records in Memory_Store, THEN THE
   Study_Companion SHALL return an empty result with a message indicating no data exists for
   that subject.

---

### Requirement 10: Session Context and Continuity

**User Story:** As a student, I want the Study_Companion to remember the context of our current
conversation, so that I don't have to repeat myself within a session.

#### Acceptance Criteria

1. WHILE a Session is active, THE Study_Companion SHALL maintain in-memory context of the
   current subject, topic, and quiz state so the Student can refer to them without restating.
2. WHEN a Session ends, THE Study_Companion SHALL flush all pending Quiz_Records and
   Study_Session_Records to Memory_Store before terminating.
3. WHEN a new Session begins, THE Study_Companion SHALL retrieve the Student's most recent
   Study_Session_Record and last 5 Quiz_Records from Memory_Store to prime the session context.
4. IF a Session is interrupted without a clean shutdown, THEN THE Study_Companion SHALL recover
   any unsaved records from a local write-ahead buffer on the next Session start and persist them
   to Memory_Store.

---

### Requirement 11: Configuration and Onboarding

**User Story:** As a developer deploying the Study_Companion, I want a clear configuration
interface, so that I can connect Hindsight, set thresholds, and go live quickly.

#### Acceptance Criteria

1. THE Study_Companion SHALL read all required configuration (Hindsight endpoint, API key,
   Weak_Area_Threshold, LLM provider, LLM model) from environment variables defined in a
   `.env.local` file.
2. IF any required configuration value is missing at startup, THEN THE Study_Companion SHALL log
   a descriptive error naming the missing variable and refuse to start the server.
3. THE Web_UI SHALL expose an onboarding flow that guides a new Student through providing their
   name, subjects, and first Exam_Event, and persists the onboarding data to Memory_Store.
4. WHERE the Weak_Area_Threshold is not configured, THE Study_Companion SHALL default to 60%.
5. THE Study_Companion SHALL include a README with step-by-step setup instructions covering:
   Hindsight Cloud registration, environment variable configuration, and running the Next.js
   development and production server.

### Requirement 12: Optional OpenClaw Skill Integration

**User Story:** As a student who prefers messaging apps, I want to optionally interact with the
Study_Companion via WhatsApp, Telegram, or Slack, so that I can use the assistant in the
communication channel I already have open.

#### Acceptance Criteria

1. WHERE OpenClaw integration is enabled, THE Study_Companion SHALL expose an HTTP webhook
   endpoint that conforms to the OpenClaw skill protocol, allowing an OpenClaw Gateway to
   forward messages from any supported channel.
2. WHERE OpenClaw integration is enabled, THE Study_Companion SHALL register itself with the
   OpenClaw Gateway using credentials supplied via environment variables.
3. WHEN a Student sends a natural-language message through an OpenClaw channel, THE
   Study_Companion SHALL parse the intent and route it to the appropriate sub-component
   (Quiz_Engine, Mistake_Tracker, Schedule_Manager, or Plan_Generator) within 3 seconds.
4. THE Study_Companion SHALL support the following explicit commands when operating as an
   OpenClaw_Skill: `/quiz`, `/mistakes`, `/plan`, `/log-session`, `/add-exam`, `/reminders`,
   `/summary`.
5. WHEN Study_Companion produces a response longer than 4000 characters via an OpenClaw channel,
   THE Study_Companion SHALL split the response into multiple messages and deliver them
   sequentially.
6. IF OpenClaw integration environment variables are absent, THEN THE Study_Companion SHALL start
   normally as a standalone web application without attempting OpenClaw registration.
7. THE Study_Companion SHALL share the same Memory_Store regardless of whether the Student
   interacts via the Web_UI or an OpenClaw channel, so that history is unified across interfaces.
